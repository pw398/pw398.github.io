# Intro

The ever-ubiquitous SQL language has been the standard for database-querying since the 70's, though numerous versions exist. In this article we will be looking at MySQL, for which the community version is free. Similar alternatives include SQL Server Express, PostgreSQL, and MariaDB. "NoSQL" languages, such as those relying on JavaScript Object Notation (JSON), have risen in popularity within the past couple of decades, and will be addressed in a future article.

Similarly, Python has become one of the most widely adopted programming languages, for tasks that don't require low-level systems such as C++. IPython notebooks (.ipynb) are extremely popular because of:
- Their shareability, readability, and transparency
- Their ability to document, provide, and preserve output at various stages of a workflow, and
- Their capability for real-time feedback while coding, which is particularly useful for debugging

IPython notebooks allow for neat features such as rendering a webpage from the workbook contents, integration with shell commands to the virtual environment, and the ability to create and use cell "magic commands", one use of which is to integrate other languages like SQL. There are also functions within various modules which will accept an SQL query in text, but don't provide the same kind of keyword-formatting you see in an SQL editor.

We'll be working in a Google Colab Python notebook, starting with shell commands, and then progressing to use <code>mysql-server</code>, <code>mysql-connector-python</code>, and <code>pymysql</code>. You should be able to recreate the functionality in your shell, .py script, etc., though various installations and configurations are required.

Of course, hosting your business's database on a Google Colab notebook would be highly problematic, as the data would only be retained for the duration of the session (or require being written to file). But it will do nicely for the sake of code comprehension.

I must also mention that it is not a good practice to include passwords in your code, as I have done for the sake of transparency. One alternative is to store those commands in a secure .sql file, and have the command reference that.

Creating a <code>.my.cnf</code> file prevents us from having to specify our username and password each time we enter shell commands. The user is set to "root" and the password to "pw". <code>-f</code> stands for "force", meaning do not prompt for confirmation, and ignore nonexistent file, and <code>-e</code> stands for "execute".

```bash
# Create .my.cnf for password-based authentication
!rm -f ~/.my.cnf /root/.my.cnf                                     # clear if existing
!echo -e "[client]\nuser=root\npassword=pw" > ~/.my.cnf            # print text to CLI
!chmod 600 ~/.my.cnf                                               # grants read/write permissions to file owner
```

# Interacting with .txt. and .csv Files

To demonstrate how we would import the database from a .txt file, I will first export classicmodels to .txt. <code>sed</code> stands for Stream Editor, a Unix-based tool for searching, finding and replacing, inserting, or deleting text in files or streams. <code>-i</code> tells <code>sed</code> to edit the file directly, rather than just printing the result to screen.

```bash
!mysqldump -u root -ppw classicmodels > text_db.txt
!sed -i 's/`classicmodels`/`text_db`/g' text_db.txt
```

Feel free to open the .txt file and inspect the format. Next, we will use the appropriate commands to import that data into a database called text_db. <code>-ppw</code> is <code>-p</code> for password, followed by our password, <code>pw</code> Unlike <code>-u root</code> for specifying our username, no space character is expected between the <code>-p</code> and the password.

```bash
!mysql -u root -ppw -e "CREATE DATABASE text_db;"
!mysql -u root -ppw text_db < text_db.txt
```

And now if we run <code>SHOW DATABASES</code>, we see text_db included.

What about .csv? Well, we cannot readily export an entire database to a .csv file, but we can work with individual tables, and loop through them if we want to address all tables. For this, we'll use <code>pymysql</code>, <code>sqlalchemy</code>, and <code>pandas</code>.

```bash
!pip install pymysql > /dev/null 2>&1
```

```python
from sqlalchemy import create_engine, text
import pandas as pd
import os

# connection setup
engine = create_engine(f"mysql+pymysql://root:pw@localhost/classicmodels")

# check if 'csv_db' folder exists, create if not
output_dir = 'csv_db'
if not os.path.exists(output_dir):
    os.makedirs(output_dir)
    print(f"Created directory: {output_dir}")

# get list of tables
with engine.connect() as conn:
    result = conn.execute(text("SHOW TABLES;"))
    tables = [row[0] for row in result]

# loop through tables and export each to the 'csv_db' directory
for table in tables:
    df = pd.read_sql(f"SELECT * FROM `{table}`", engine)
    file_path = os.path.join(output_dir, f"{table}.csv")
    df.to_csv(file_path, index=False)

print(f"Export complete. All CSV files are in the '{output_dir}' directory.")
```
